# How do we know what an LLM's response looks like?

**Question asked:** How do we know that these are the datatypes or arguments
for these classes (`Message` / `ToolCall` / `ModelResponse` in
`providers/base.py`)? How do we know what to expect from an LLM to return?

---

## Two different "shapes," two different sources of truth

**1. `providers/base.py`'s dataclasses (`Message`, `ToolCall`,
`ModelResponse`) are *not* dictated by the LLM at all.** Those are shapes
*we* invented, for our own convenience, so the rest of the codebase never has
to think about which provider it's talking to. The LLM has never heard of
`ModelResponse`.

**2. What actually comes back over the wire *is* fixed — but it's fixed by
the provider's published API contract**, not something you infer by trial
and error. Since this project talks to Ollama through the `openai` Python
SDK (Ollama exposes an OpenAI-compatible endpoint), the response shape is
defined by that SDK's own type definitions — literally installed in this
repo's `.venv` at
`.venv/lib/python3.11/site-packages/openai/types/chat/chat_completion.py`:

```python
class Choice(BaseModel):
    finish_reason: Literal["stop", "length", "tool_calls", "content_filter", "function_call"]
    """The reason the model stopped generating tokens.

    This will be `stop` if the model hit a natural stop point or a provided stop
    sequence, `length` if the maximum number of tokens specified in the request was
    reached, `content_filter` if content was omitted due to a flag from our content
    filters, `tool_calls` if the model called a tool, or `function_call`
    (deprecated) if the model called a function.
    """

    index: int
    logprobs: Optional[ChoiceLogprobs] = None
    message: ChatCompletionMessage
    """A chat completion message generated by the model."""
```

This isn't guesswork — `finish_reason` can *only* ever be one of those five
exact strings, because OpenAI defined it as a `Literal[...]` and every
provider that claims "OpenAI-compatible" (including Ollama) commits to
returning one of them.

## Following one field all the way through

Take `tool_calls`. The chain of custody, from
`.venv/lib/python3.11/site-packages/openai/types/chat/chat_completion_message_function_tool_call.py`:

```python
class Function(BaseModel):
    arguments: str
    """
    The arguments to call the function with, as generated by the model in JSON
    format. Note that the model does not always generate valid JSON, and may
    hallucinate parameters not defined by your function schema. Validate the
    arguments in your code before calling your function.
    """

    name: str
    """The name of the function to call."""


class ChatCompletionMessageFunctionToolCall(BaseModel):
    id: str
    """The ID of the tool call."""

    function: Function
    """The function that the model called."""

    type: Literal["function"]
    """The type of the tool. Currently, only `function` is supported."""
```

This is the piece that matters most for understanding our `ToolCall`
dataclass:

```
response.choices[0].message.tool_calls[i]   ← SDK's shape, defined above
    .id            → str
    .function.name → str
    .function.arguments → str  (a JSON-encoded STRING, not a dict!)
```

Notice `arguments: str`, with a docstring warning: *"the model does not
always generate valid JSON, and may hallucinate parameters... validate
before calling."* The model outputs raw text that's supposed to look like
`{"path": "requirements.txt"}` — but it's still just text until you parse
it. That's why `providers/local.py` (per `docs/AGENT_ARCHITECTURE.md`'s
reference implementation) has to do:

```python
arguments=json.loads(tc.function.arguments)
```

before it can go into our own `ToolCall.arguments: dict[str, Any]`. Our
dataclass stores it as an already-parsed dict because *we* decided that's
more convenient to work with — the SDK never hands you a dict directly, you
convert it yourself. And `dict[str, Any]` is provably correct as a type, not
just convenient: JSON objects are always string-keyed, so parsing any valid
JSON object always produces `dict[str, ...]` — the `Any` just admits the
value can be a string, number, list, nested object, etc.

## The mapping, end to end

| SDK gives you | → | Our dataclass |
|---|---|---|
| `response.choices[0].message.content` | → | `ModelResponse.content` |
| `response.choices[0].message.tool_calls` (parsed one by one) | → | `ModelResponse.tool_calls: list[ToolCall]` |
| `response.choices[0].finish_reason` | → | `ModelResponse.stop_reason` |
| `response.usage` | → | `ModelResponse.usage` |

`providers/local.py` is the *only* file that's supposed to know this mapping
exists. Once it's implemented, nothing else in the codebase ever touches
`response.choices[0].message.tool_calls` directly — everything downstream
only sees our own `ModelResponse`.

## So, in short

You know what to expect because:

1. The SDK you depend on (`openai`, pinned in `pyproject.toml`) ships typed
   Pydantic models describing exactly what its methods return — you can
   read the source directly, as above.
2. Ollama's `/v1/chat/completions` endpoint is documented as
   OpenAI-compatible, so it commits to returning that same shape.
3. Your own dataclasses (`Message`/`ToolCall`/`ModelResponse`) are a
   separate, self-imposed contract — one you get to design however's
   convenient, precisely so the rest of your code doesn't need to know any
   of the above.

If you ever add a second provider (Anthropic, say), its SDK has *different*
typed classes (`content` is a list of blocks instead of a string, tool calls
are `tool_use` blocks, arguments come back already parsed as a dict, not a
string) — that provider's file would do its own translation into the
*same* `ModelResponse` shape. That's the entire payoff of the provider
abstraction described in `docs/AGENT_ARCHITECTURE.md` §7.
