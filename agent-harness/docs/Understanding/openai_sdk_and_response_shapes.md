# Why We Use the OpenAI SDK, and How We Know What an LLM Returns

A running Q&A thread, kept in order as asked, starting from the original
question about `providers/base.py`'s dataclasses.

---

## Q1: "How do we know that these are the datatypes or arguments for these classes? I don't understand how we know what to expect from an LLM to return."

### Two different "shapes," two different sources of truth

**1. `providers/base.py`'s dataclasses (`Message`, `ToolCall`,
`ModelResponse`) are *not* dictated by the LLM at all.** Those are shapes
*we* invented, for our own convenience, so the rest of the codebase never
has to think about which provider it's talking to. The LLM has never heard
of `ModelResponse`.

**2. What actually comes back over the wire *is* fixed — but it's fixed by
the provider's published API contract**, not something you infer by trial
and error. Since this project talks to Ollama through the `openai` Python
SDK (Ollama exposes an OpenAI-compatible endpoint), the response shape is
defined by that SDK's own type definitions — literally installed in this
repo's `.venv` at
`.venv/lib/python3.11/site-packages/openai/types/chat/chat_completion.py`:

```python
class Choice(BaseModel):
    finish_reason: Literal["stop", "length", "tool_calls", "content_filter", "function_call"]
    """The reason the model stopped generating tokens.

    This will be `stop` if the model hit a natural stop point or a provided stop
    sequence, `length` if the maximum number of tokens specified in the request was
    reached, `content_filter` if content was omitted due to a flag from our content
    filters, `tool_calls` if the model called a tool, or `function_call`
    (deprecated) if the model called a function.
    """

    index: int
    logprobs: Optional[ChoiceLogprobs] = None
    message: ChatCompletionMessage
    """A chat completion message generated by the model."""
```

This isn't guesswork — `finish_reason` can *only* ever be one of those five
exact strings, because OpenAI defined it as a `Literal[...]` and every
provider that claims "OpenAI-compatible" (including Ollama) commits to
returning one of them.

### Following one field all the way through

Take `tool_calls`. The chain of custody, from
`.venv/lib/python3.11/site-packages/openai/types/chat/chat_completion_message_function_tool_call.py`:

```python
class Function(BaseModel):
    arguments: str
    """
    The arguments to call the function with, as generated by the model in JSON
    format. Note that the model does not always generate valid JSON, and may
    hallucinate parameters not defined by your function schema. Validate the
    arguments in your code before calling your function.
    """

    name: str
    """The name of the function to call."""


class ChatCompletionMessageFunctionToolCall(BaseModel):
    id: str
    """The ID of the tool call."""

    function: Function
    """The function that the model called."""

    type: Literal["function"]
    """The type of the tool. Currently, only `function` is supported."""
```

The piece that matters for our `ToolCall` dataclass:

```
response.choices[0].message.tool_calls[i]   ← SDK's shape, defined above
    .id            → str
    .function.name → str
    .function.arguments → str  (a JSON-encoded STRING, not a dict!)
```

Notice `arguments: str`, with a docstring warning: *"the model does not
always generate valid JSON, and may hallucinate parameters... validate
before calling."* The model outputs raw text that's supposed to look like
`{"path": "requirements.txt"}` — but it's still just text until you parse
it. That's why `providers/local.py` (per `docs/AGENT_ARCHITECTURE.md`'s
reference implementation) has to do:

```python
arguments=json.loads(tc.function.arguments)
```

before it can go into our own `ToolCall.arguments: dict[str, Any]`. Our
dataclass stores it as an already-parsed dict because *we* decided that's
more convenient to work with — the SDK never hands you a dict directly, you
convert it yourself. And `dict[str, Any]` is provably correct as a type, not
just convenient: JSON objects are always string-keyed, so parsing any valid
JSON object always produces `dict[str, ...]` — the `Any` just admits the
value can be a string, number, list, nested object, etc.

### The mapping, end to end

| SDK gives you | → | Our dataclass |
|---|---|---|
| `response.choices[0].message.content` | → | `ModelResponse.content` |
| `response.choices[0].message.tool_calls` (parsed one by one) | → | `ModelResponse.tool_calls: list[ToolCall]` |
| `response.choices[0].finish_reason` | → | `ModelResponse.stop_reason` |
| `response.usage` | → | `ModelResponse.usage` |

`providers/local.py` is the *only* file that's supposed to know this mapping
exists. Once it's implemented, nothing else in the codebase ever touches
`response.choices[0].message.tool_calls` directly — everything downstream
only sees our own `ModelResponse`.

### So, in short

You know what to expect because:

1. The SDK you depend on (`openai`, pinned in `pyproject.toml`) ships typed
   Pydantic models describing exactly what its methods return — you can
   read the source directly, as above.
2. Ollama's `/v1/chat/completions` endpoint is documented as
   OpenAI-compatible, so it commits to returning that same shape.
3. Your own dataclasses (`Message`/`ToolCall`/`ModelResponse`) are a
   separate, self-imposed contract — one you get to design however's
   convenient, precisely so the rest of your code doesn't need to know any
   of the above.

## Q2: "Are these SDKs free? And why do we need these SDKs — why OpenAI or Anthropic SDKs?"

### Are the SDKs free?

Yes — the SDK itself (`pip install openai`, or `anthropic`) is a free,
open-source Python package (Apache-2.0 license, confirmed via the installed
package metadata). Installing and importing it costs nothing.

**What costs money is not the SDK — it's the API calls it makes.** If you
point the `openai` SDK at OpenAI's actual servers with a real API key, you
pay per token for their hosted models. But this project points it at
`base_url="http://localhost:11434/v1"` — your own local Ollama server.
`api_key="ollama"` in the code is a dummy value; the SDK requires *some*
non-empty string in that field, but Ollama doesn't check it. So: SDK free,
and in this project's case, the calls it makes are free too, because
there's no paid service on the other end.

Only `openai` is listed in `pyproject.toml` — there's no `anthropic`
dependency in this project yet.

### Why do we need an SDK at all, and why *this* one?

You technically don't *need* one — under the hood, calling a model is just
an HTTP POST with a JSON body, and you get JSON back. You could do that
with `requests` or `httpx` directly, no SDK required.

The SDK is a convenience layer that saves you from hand-rolling:
- request/response serialization
- retries, timeouts, connection pooling
- streaming (parsing server-sent events token-by-token)
- **typed response objects** — this is what let us read
  `Choice.finish_reason: Literal["stop", "tool_calls", ...]` and
  `Function.arguments: str` directly from source in Q1, instead of guessing
  at raw JSON keys

**Why the `openai` SDK specifically, given this project doesn't use
OpenAI's cloud at all:** Ollama deliberately implements an OpenAI-compatible
endpoint (`/v1/chat/completions`) — it copies OpenAI's wire format on
purpose so that any client built for OpenAI's API works against it
unmodified. That means this project gets a mature, typed, well-tested HTTP
client for free, just by pointing it at `localhost` instead of
`api.openai.com`. Writing that client from scratch would mean
re-implementing all the plumbing above just to reach your own laptop.

**Where Anthropic's SDK would come in:** only if/when this project adds
Anthropic as a second provider (the `provider: "local" | "anthropic" |
"openai"` field sketched in `docs/AGENT_ARCHITECTURE.md`'s `config.py`).
Anthropic's Messages API has a genuinely different shape (content blocks
instead of a flat string, `tool_use` blocks instead of `tool_calls`,
arguments come back already parsed as a dict, not a JSON string) — so it
isn't OpenAI-compatible, and you'd need Anthropic's own SDK (or a
hand-rolled `httpx` client) to talk to it. At that point, calling
Anthropic's *actual cloud models* would cost real money per token — the SDK
stays free, the inference doesn't.

## Q3: "Explain like I'm 5 — what do you mean by 'wire format'? What is an 'OpenAI-compatible endpoint'?"

### "Wire format" = the exact shape of what you send/get back

Imagine two people talking through walkie-talkies. For them to understand
each other, they need to agree on things like: "say 'over' when you're done
talking," "start with your call sign," etc. That agreed-upon set of rules —
how the message is packaged — is the "format."

When your Python code talks to a model server, it's not talking in English.
It's sending a blob of text (JSON) over the internet wire, shaped exactly
like this:

```
{"model": "llama3.1", "messages": [{"role": "user", "content": "hi"}]}
```

...and it expects to get a blob of text back shaped exactly like this:

```
{"choices": [{"message": {"content": "hello!"}, "finish_reason": "stop"}]}
```

**That exact shape — these specific field names, nested this specific
way — is the "wire format."** It's called that because it's literally what
travels across the wire (the network cable/wifi) between the two computers.

### "OpenAI-compatible" = someone else copied OpenAI's shape on purpose

OpenAI was the first big company to popularize this exact shape
(`messages`, `choices`, `finish_reason`, etc.) for talking to a chat model.
Lots of other companies built their own model servers *after* OpenAI. They
had a choice:

- Invent their own new shape, forcing everyone to write new code to talk to
  them, **or**
- Copy OpenAI's shape exactly, so anyone who already wrote code for OpenAI
  can talk to them for free

Ollama chose door #2. They built their own server (running on your
laptop), but they deliberately made it accept and reply with the *exact
same JSON shape* OpenAI uses. That's what "OpenAI-compatible" means —
nothing more mystical than "we copied their form."

### Why that matters for the SDK

Think of the `openai` SDK as a pre-printed order form + a mail carrier,
bundled together. It already knows:
- exactly how to fill out the form (build that JSON)
- exactly where to mail it (send an HTTP request)
- exactly how to read the reply that comes back

Normally that form only gets mailed to one address: `api.openai.com`
(OpenAI's real servers, which cost money per request).

But the SDK lets you change *just the address* it mails to:

```python
AsyncOpenAI(base_url="http://localhost:11434/v1", ...)
#                     ^^^^^^^^^^^^^^^^^^^^^^^^^^^ your own laptop, not OpenAI
```

Since Ollama is sitting at that address on your laptop, and Ollama agreed
to accept mail in the exact same form OpenAI uses, **the same pre-printed
form works** — you didn't have to design a new form or hire a new mail
carrier. You just changed the address on the envelope.

That's the whole trick: same envelope, same handwriting, different mailbox.

## Q4: What exactly does `ChatCompletion` return, and how does that map to `base.py` / `local.py`?

### The full shape, top to bottom

`await self.client.chat.completions.create(...)` (called in
`providers/local.py`'s `complete()`) returns one `ChatCompletion` object.
Here is its entire tree, straight from the SDK source in
`.venv/lib/python3.11/site-packages/openai/types/chat/`:

```
ChatCompletion                                     (chat_completion.py)
├── id: str
├── object: Literal["chat.completion"]
├── created: int
├── model: str
├── service_tier: Optional[Literal[...]]
├── system_fingerprint: Optional[str]
├── choices: List[Choice]                          ← usually just one
│   └── Choice
│       ├── index: int
│       ├── finish_reason: Literal["stop", "length", "tool_calls",
│       │                          "content_filter", "function_call"]
│       ├── logprobs: Optional[ChoiceLogprobs]
│       └── message: ChatCompletionMessage         (chat_completion_message.py)
│           ├── role: Literal["assistant"]
│           ├── content: Optional[str]
│           ├── refusal: Optional[str]
│           ├── annotations, audio, function_call  ← unused here
│           └── tool_calls: Optional[List[ChatCompletionMessageToolCallUnion]]
│               └── ChatCompletionMessageFunctionToolCall
│                                       (chat_completion_message_function_tool_call.py)
│                   ├── id: str
│                   ├── type: Literal["function"]
│                   └── function: Function
│                       ├── name: str
│                       └── arguments: str          ← JSON-encoded, not a dict
└── usage: Optional[CompletionUsage]                (completion_usage.py)
    ├── prompt_tokens: int
    ├── completion_tokens: int
    ├── total_tokens: int
    ├── prompt_tokens_details: Optional[...]
    └── completion_tokens_details: Optional[...]
```

### Line-by-line: how `providers/local.py` turns that into `providers/base.py`'s shapes

This is the actual implementation currently in the repo:

```python
response = await self.client.chat.completions.create(   # → ChatCompletion
    model=self.model,
    messages=self._format_messages(messages, system),
    tools=self._format_tools(tools) if tools else None,
)
msg = response.choices[0].message                        # → ChatCompletionMessage

tool_calls = []
if msg.tool_calls:
    tool_calls = [
        ToolCall(
            id=tc.id,                                     # .id            → ToolCall.id
            name=tc.function.name,                         # .function.name → ToolCall.name
            arguments=json.loads(tc.function.arguments),   # .function.arguments (str) → parsed → ToolCall.arguments (dict)
        )
        for tc in msg.tool_calls
    ]

return ModelResponse(
    content=msg.content or "",                             # ChatCompletionMessage.content → ModelResponse.content
    tool_calls=tool_calls,                                 # built above          → ModelResponse.tool_calls
    stop_reason=response.choices[0].finish_reason,         # Choice.finish_reason → ModelResponse.stop_reason
    usage={
        "input_tokens": response.usage.prompt_tokens,      # CompletionUsage.prompt_tokens     → our usage dict
        "output_tokens": response.usage.completion_tokens, # CompletionUsage.completion_tokens  → our usage dict
    },
)
```

A table view of the same mapping:

| From `ChatCompletion` | Field | → | `base.py` shape | Field |
|---|---|---|---|---|
| `response` | `choices[0].message.tool_calls[i].id` | → | `ToolCall` | `.id` |
| `response` | `choices[0].message.tool_calls[i].function.name` | → | `ToolCall` | `.name` |
| `response` | `choices[0].message.tool_calls[i].function.arguments` (str, needs `json.loads`) | → | `ToolCall` | `.arguments` (dict) |
| `response` | `choices[0].message.content` | → | `ModelResponse` | `.content` |
| — | (list of `ToolCall`s above) | → | `ModelResponse` | `.tool_calls` |
| `response` | `choices[0].finish_reason` | → | `ModelResponse` | `.stop_reason` |
| `response` | `usage.prompt_tokens` | → | `ModelResponse` | `.usage["input_tokens"]` |
| `response` | `usage.completion_tokens` | → | `ModelResponse` | `.usage["output_tokens"]` |

### What gets thrown away, and why that's fine

Notice how much of the `ChatCompletion` tree above is **never touched**:
`id`, `created`, `model`, `object`, `service_tier`, `system_fingerprint`,
`logprobs`, `refusal`, `annotations`, `audio`, the deprecated
`function_call`, `total_tokens`, and both `*_details` breakdowns. None of
that gets copied into `ModelResponse`.

That's not an oversight — it's the point of the provider layer. `local.py`
isn't just translating field names, it's **filtering down to only what the
agent loop actually needs to keep running.** `ModelResponse` is deliberately
smaller than `ChatCompletion` because the loop only ever asks three
questions: *what did it say, does it want a tool, why did it stop.*
Anything else is dropped at the boundary, on purpose, so it never leaks into
the provider-agnostic side of the codebase.

Also worth noticing: `input_tokens` / `output_tokens` are *our* chosen key
names, not OpenAI's (`prompt_tokens` / `completion_tokens`). That renaming
happens for the same reason as everything else in `base.py` — it's the
vocabulary Anthropic's API already uses for the same concept, so if a
second provider is added later, both providers can fill in the same two
keys without either one needing to know the other's native field names.

### One more shape, mentioned for completeness: streaming

`stream()` in `local.py` calls the same endpoint with `stream=True`, which
returns a different SDK type per chunk — `ChatCompletionChunk`, not
`ChatCompletion` — where each chunk exposes `choices[0].delta.content`
(a small incremental piece of text) instead of a full `message`. That's why
`stream()`'s loop reads `chunk.choices[0].delta.content` rather than
`.message.content` — it's a related but distinct shape, documented in
`chat_completion_chunk.py` in the same SDK directory.
